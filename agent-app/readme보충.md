앞서 깊게 파고들었던 `useradd`의 옵션 해석과 `chown root:agent-core`의 커널 레벨 보안(금고 비유) 내용을 세 번째 파트(그룹 관리와 ACL)의 흐름에 맞게 완벽히 녹여내어 하나의 통합된 가이드로 완성해 드립니다.

복사해서 바로 문서화하시기 좋도록 정리했습니다.

### 첫 번째 질문 묶음: Docker (도커)

**1. `--privileged` 옵션은 무엇인가요?**

- **직관적 설명:** 컨테이너에게 "이 컴퓨터(호스트)의 진짜 관리자(Root) 권한"을 빌려주는 옵션입니다.
- **왜 쓰는가:** 원래 도커 컨테이너는 보안을 위해 시스템의 핵심적인 부분(네트워크 딥한 설정, 방화벽 제어 등)을 건드리지 못하도록 격리되어 있습니다. 하지만 이번 실습에서는 컨테이너 안에서 직접 방화벽(`ufw`)도 켜고 시스템 설정도 바꿔야 하므로, 예외적으로 무소불위의 권한(`privileged`)을 부여한 것입니다.

**2. `-p 20022:20022` 땡땡(:)의 의미는 무엇인가요?**

- **직관적 설명:** `내 진짜 컴퓨터 포트 : 도커 안의 가상 컴퓨터 포트`를 연결(포트 포워딩)해주는 기호입니다.
- **왜 두 번 쓰는가:** 왼쪽은 '내 윈도우/맥 PC의 20022번 포트'이고, 오른쪽은 '도커 컨테이너 내부의 20022번 포트'입니다. 즉, "누군가 내 PC의 20022번으로 찾아오면, 도커 안의 20022번으로 토스해 줘!"라는 문지기 역할을 설정하는 것입니다. 15034번도 마찬가지 원리입니다.

### 두 번째 질문 묶음: SSH 설정, `sed`, 방화벽

**1. `sed` 코드가 왜 이렇게 생겼고, 각 알파벳은 무슨 뜻인가요?**
이건 리눅스에서 텍스트 에디터(vi, 메모장)를 열지 않고, 명령어로 '찾아서 바꾸기'를 하는 마법의 주문입니다.

- `sudo`: 최고 관리자 권한으로 실행해라.
- `sed`: Stream Editor의 약자. 텍스트를 조작하는 프로그램입니다.
- `-i`: in-place의 약자. 출력만 하지 말고 **실제 원본 파일에 덮어써서 저장해라**.
- `'s/A/B/g'`: 여기가 핵심 공식입니다.
- `s` (substitute): 교체 작업을 시작하겠다.
- `/`: 그냥 구역을 나누는 빗장(구분자)입니다.
- `A` (찾을 단어): `#Port 22` (주석 처리된 22번 포트 설정)
- `B` (바꿀 단어): `Port 20022` (새로운 포트 설정)
- `g` (global): 파일 전체를 뒤져서 전부 다 바꿔라.

- `/etc/ssh/sshd_config`: 최종적으로 조작할 대상 파일의 경로입니다.

**2. 아까는 `ss` 쓰라더니 여기선 왜 `netstat`을 쓰나요?**

- **정확한 지적입니다!** 이전 질문에서 분석하셨듯, 2026년 기준 최신 리눅스에서는 `ss`를 쓰는 것이 맞습니다.
- **이유:** 이 교안이나 스크립트를 작성한 사람이 과거의 레거시 습관(`netstat`)을 그대로 가져다 쓴 것입니다. 현업이었다면 `sudo ss -tulnp | grep ssh`로 쓰는 것이 훨씬 세련되고 속도도 빠릅니다. 아주 날카롭게 잘 짚어내셨습니다.

**3. '기본 인바운드 차단(default deny incoming)'이 뭔가요?**

- 방화벽의 가장 기본 원칙인 **화이트리스트(Zero Trust) 방식**을 선언하는 것입니다.
- "일단 외부에서 우리 서버로 들어오는(incoming) 모든 접속은 묻지도 따지지도 않고 다 막아(deny)!"라고 철벽을 치는 겁니다. 그 다음에 내가 허락할 포트(20022, 15034)만 핀셋으로 구멍을 뚫어주는(`allow`) 것이 가장 안전하기 때문입니다.

**4. 왜 `tcp`로 허용해주나요?**

- 인터넷 통신 방식에는 꼼꼼한 `TCP`와 빠르지만 막무가내인 `UDP`가 있습니다.
- SSH 접속이나 웹 서비스(HTTP)는 데이터를 주고받을 때 순서가 꼬이거나 데이터가 유실되면 치명적입니다. 따라서 서로 연결을 확인하고 데이터를 꼼꼼하게 주고받는 **TCP(Transmission Control Protocol)** 규약을 사용하기 때문에 TCP만 허용하는 것입니다.

**5. `(v6)`와 그냥의 차이는 무엇인가요?**

- **그냥 (예: 20022/tcp):** 우리가 흔히 아는 기존 인터넷 주소 체계인 **IPv4** (예: `192.168.0.1`)를 통한 접속을 허용한다는 뜻입니다.
- **(v6):** 인터넷 주소가 고갈되어서 새로 만든 길고 복잡한 주소 체계인 **IPv6** (예: `2001:0db8:85a3...`)를 통한 접속을 허용한다는 뜻입니다.
- UFW 방화벽은 똑똑해서 명령어를 한 번만 쳐도 두 가지 버전의 인터넷 주소 체계를 모두 열어줍니다.

### 세 번째 질문 묶음: 그룹 관리와 권한 제어(ACL)

**1. 계정들을 왜 이렇게 그룹으로 묶나요?**

- 역할 기반 권한 제어(RBAC)를 위해서입니다.
- 회사에 비유하면 `agent-common`은 '전 직원 그룹'이고, `agent-core`는 '핵심 개발팀'입니다.
- 테스터(`agent-test`)는 앱이 잘 돌아가는지 확인만 하면 되므로 '전 직원' 그룹에만 넣습니다. 반면 관리자와 개발자(`agent-admin`, `agent-dev`)는 서버를 수정하고 보안 키를 만져야 하므로 '핵심 개발팀' 그룹에도 소속을 시킨 것입니다. 이렇게 해야 나중에 폴더 권한을 줄 때 "핵심 개발팀만 들어오세요"라고 한 방에 통제할 수 있습니다.

**2. 계정 생성 명령어(`useradd`)는 어떻게 구성되어 있나요?**
단순히 계정만 만드는 게 아니라 방, 터미널, 소속을 한 번에 세팅하는 실무 명령어입니다. (`sudo useradd -m -s /bin/bash -G agent-common,agent-core agent-admin`)

- **`-m` (Make home):** 새 사용자의 개인 방(홈 디렉토리)을 자동으로 만들어 줍니다.
- **`-s /bin/bash` (Shell):** 로그인 시 사용할 기본 터미널을 리눅스 표준인 `bash`로 지정합니다.
- **`-G` (Groups):** 보조 그룹을 지정합니다. 쉼표로 띄어쓰기 없이 적으면 두 그룹에 동시 가입됩니다.

**3. 소유권 변경(`chown root:agent-core`) 코드의 진짜 의미는 무엇인가요?**
이 코드는 단순 소유권 변경을 넘어, 해커의 시스템 파괴와 증거 인멸을 막는 커널 레벨의 방어막(금고 설계)입니다.

- **`root` (개인 소유자 = 건물주):** 리눅스 커널에서 권한 검사를 패스하는 절대 권력자(UID 0)입니다. 폴더의 껍데기(존재 자체)를 지울 수 있는 권한을 `root`에게 독점시킵니다. 해커가 일반 관리자 계정을 털어도 치명적인 `api_keys` 폴더나 CCTV 역할을 하는 `로그 폴더` 자체를 삭제(`rm -rf`)하지 못하게 막아냅니다.
- **`:` (콜론):** "앞은 개인 주인(존재 통제), 뒤는 그룹(작업 통제)"을 나누는 절대 규칙입니다. 순서가 바뀌면 방어막이 완전히 역전되어 붕괴합니다.
- **`agent-core` (소유 그룹 = 세입자 동아리):** 이 폴더 안에 들어가서 파일을 읽고 쓰는 '작업 출입증'만 부여받습니다.
- **결론:** 내용물(서류)은 운영팀이 만지게 해주되, 금고(폴더) 자체는 최고 관리자만 부술 수 있게 박아두는 완벽한 역할 분리입니다.

**4. ACL이 뭐고 왜 자물쇠라고 부르나요?**

- **ACL (Access Control List - 접근 제어 목록):** 기존 리눅스 권한 시스템의 한계를 극복하는 '디지털 VIP 패스'입니다.
- **기존 권한의 한계:** 원래 리눅스는 파일에 권한을 줄 때 딱 3칸(소유자, 소유그룹, 나머지 전부)밖에 없습니다. 세밀한 조정이 불가능합니다.
- **자물쇠인 이유:** `api_keys` 폴더는 절대 털리면 안 되는 곳입니다. 그래서 기본 권한으로 문을 잠가두고, ACL이라는 특수 자물쇠를 하나 더 달아서 "딱 `agent-core` 그룹 멤버들만 이 폴더에서 읽고/쓰고/실행할 수 있어!"라고 명시적으로 VIP 명단을 박아넣는 것입니다.

**5. ACL 코드 자세히 해석**

- `sudo setfacl -m g:agent-core:rwx /home/.../api_keys`
- `setfacl`: Set File ACL (ACL 규칙을 세팅해라)
- `-m`: modify (규칙을 추가하거나 수정하겠다)
- `g:agent-core:rwx`: **Group**(`g`) 중에서 `agent-core`라는 그룹에게, Read(`r`), Write(`w`), eXecute(`x`) 권한을 모두 주어라.

- `getfacl /home/.../api_keys`
- `getfacl`: Get File ACL (현재 이 폴더에 어떤 ACL VIP 명단이 적혀있는지 화면에 보여달라)

앞서 정리하신 문서의 마지막(3번 묶음) 바로 아래에 이어서 붙여넣으실 수 있도록, 4~6번째 질문에 대한 해설을 동일한 마크다운 양식으로 정리해 드립니다.

### 네 번째 질문 묶음: 환경 변수와 `.bashrc`

**1. 왜 `root` 계정이 아닌 `agent-admin` 계정으로 전환 후 실행해야 하는가?**

- **이유:** 환경 변수 설정 파일인 `.bashrc`는 '각 사용자마다 개인 방에 따로 존재하는 파일'이기 때문입니다.
- **비유:** `.bashrc`는 '내 방 컴퓨터의 바탕화면 설정'과 같습니다. 내가 로그인했을 때 바탕화면에 아이콘을 어떻게 배치할지 정해두는 곳이죠.
- **결과:** 만약 `root` 계정인 상태에서 저 명령어를 치면, `root`의 바탕화면( `/root/.bashrc` )에 설정이 들어갑니다. 정작 나중에 앱을 실행할 주체인 `agent-admin` 계정으로 로그인해보면 환경 변수가 하나도 적용되어 있지 않아 에러가 나게 됩니다. 따라서 반드시 "내 방(`su - agent-admin`)"으로 돌아간 뒤에 내 설정 파일(`~/.bashrc`)을 건드려야 합니다.

**2. `echo '...' >> ~/.bashrc` 가 대체 무슨 뜻인가?**

- **직관적 설명:** "왼쪽의 글자(`echo`)를 오른쪽 파일(`.bashrc`)의 맨 밑에 추가(>>)해서 저장해라"라는 뜻입니다. 편집기를 열고 타자를 치는 대신, 명령어로 파일 끝에 문장을 써넣는 기술입니다.
- `echo`: 메아리처럼 뒤에 오는 문장을 그대로 출력하라는 명령어.
- `>>`: 리다이렉션(이어쓰기). "화면에 출력하지 말고, 파일의 가장 밑줄에 이어서 써라"
- `~/.bashrc`: 내 홈 폴더(`~`) 안에 숨겨진(`.`) bash 설정 파일.

- **요약:** 텍스트 에디터(메모장)를 켜서 맨 아랫줄로 이동한 다음, 환경 변수 문장들을 타자로 쳐서 넣고 저장하는 일련의 과정을 명령어 5줄로 자동화한 것입니다.

**3. `source ~/.bashrc` 와 `env | grep AGENT` 는 무엇인가?**

- **`source ~/.bashrc`:**
- 방금 `.bashrc` 파일에 글씨를 써넣긴 했지만, 리눅스는 새로 고침을 해주기 전까지는 바뀐 내용을 모릅니다. (마치 윈도우에서 프로그램을 설치하고 '적용' 버튼을 누르거나 재부팅을 해야 하는 것과 같습니다).
- `source` 명령어는 "재부팅하지 말고, 방금 수정한 설정 파일(`.bashrc`)을 지금 당장 이 창에 다시 읽어 들여서 켜라!"라는 '즉시 적용(새로고침)' 명령어입니다.

- **`env | grep AGENT`:**
- 방금 설정한 변수들이 제대로 적용되었는지 확인하는 명령어입니다.
- `env`: 현재 내 계정에 적용된 **모든** 환경 변수(Environment)를 화면에 쏟아냅니다. (너무 많아서 보기 힘듭니다)
- `|` (파이프): 왼쪽 명령어(`env`)의 쏟아지는 결과를 오른쪽 명령어(`grep`)의 재료로 넘겨줍니다.
- `grep AGENT`: "그 수많은 변수들 중에서 이름에 'AGENT'라는 글자가 포함된 줄만 걸러서(필터링해서) 보여줘!"

- **`grep`의 유래:** Global Regular Expression Print 의 약자입니다. (전체 영역에서 정규표현식으로 찾아서 출력하라). 리눅스에서 가장 많이 쓰이는 '검색 필터' 명령어입니다.

**4. `echo "agent_api_key_test" > $AGENT_KEY_PATH/secret.key` 이건 생성이 맞나? 왜?**

- **네, 맞습니다.** 파일 생성과 동시에 내용을 써넣는 마법의 명령어입니다.
- 방금 2번에서 본 `>>` (이어쓰기)와 비슷하지만 여긴 `>` (덮어쓰기/새로 만들기)가 쓰였습니다.
- **동작 원리:** "agent_api_key_test"라는 문자를 출력하려는데, 오른쪽 경로에 `secret.key`라는 파일이 **없으면 새로 만들어서** 그 안에 글자를 넣고, 만약 **이미 있으면 기존 내용을 다 지우고** 저 글자로 덮어씁니다.
- **왜 쓰는가:** 실습용 앱이 작동하려면 특정 위치에 `secret.key`라는 파일이 반드시 있어야 하고, 그 안의 암호가 "agent_api_key_test"여야 한다고 과제 요구사항에 정해져 있기 때문입니다. 편집기를 여는 수고를 덜어준 셈입니다.

### 다섯 번째 질문 묶음: 백그라운드 실행과 리다이렉션

**1. `$AGENT_HOME/agent-app > $AGENT_LOG_DIR/agent_app.log 2>&1 &` 코드가 괴랄하게 생겼다?**
리눅스 서버 운영의 꽃이자, 초보자들을 가장 괴롭히는 **'표준 입출력 및 백그라운드 실행'** 공식입니다. 하나씩 뜯어보면 아주 논리적입니다.

- **`$AGENT_HOME/agent-app`**: 일단 이 프로그램을 실행시켜라.
- **`> $AGENT_LOG_DIR/agent_app.log`**: 프로그램이 화면에 띄우는 정상적인 메시지(표준 출력, 숫자 1번)를 화면에 보여주지 말고 저 `.log` 파일에 저장해라.
- **`2>&1`**: 여기가 괴랄해 보이는 원인입니다.
- 리눅스는 정상적인 출력물(1번, stdout)과 에러가 났을 때 내뿜는 출력물(2번, stderr)의 통로가 다릅니다.
- `2>&1`의 뜻은 "에러 메시지(2)의 방향(`>`)을, 정상 출력물(1)이 가고 있는 통로(`&1`)와 합쳐라!"입니다.
- 결과적으로 정상 로그든 에러 로그든 가리지 않고 몽땅 하나의 `.log` 파일로 몰아서 저장하게 됩니다.

- **`&` (맨 마지막 암퍼샌드)**: 이 명령어를 **백그라운드(Background)에서 돌려라!**
- 보통 프로그램을 실행하면 터미널 창을 그 프로그램이 독점해버려서 다음 명령어를 칠 수 없습니다 (예: 압축을 푸는 동안 창이 멈춰있음). 창을 끄면 프로그램도 죽습니다.
- 맨 끝에 `&`를 붙이면 "너는 눈에 안 보이는 뒷단(백그라운드)에서 조용히 돌아가고, 내 터미널 창은 계속 다른 명령어를 칠 수 있게 반환해 줘!"라는 뜻이 됩니다. 서버에서 앱을 켤 때 무조건 쓰는 방식입니다.

### 여섯 번째 질문 묶음: 다시 돌아온 `chown`

**1. `chown agent-admin:agent-core /home/agent-admin/agent-app` 은 아까 `root` 때랑 왜 다른가?**
정말 눈썰미가 좋으십니다. 앞서 세 번째 질문에서 배웠던 보안 정책(`root:agent-core`)의 연장선이자, 반대되는 상황입니다.

- **이전 상황 (`api_keys` 보안 폴더):** 그 폴더는 무조건 지켜야 하니까 방어력을 최대로 올려 건물주(`root`) 명의로 박았습니다.
- **현재 상황 (`agent-app` 작업 폴더):** 이 폴더는 개발자인 `agent-admin`이 마음대로 요리하고 지지고 볶아야 하는 일반적인 '작업 공간'입니다.
- **의도:** "이 폴더의 주인은 `agent-admin`(본인)으로 인정해 줄게. 네 마음대로 폴더 안에 `bin` 같은 새 폴더도 만들고 지워라. 단, 관리 그룹은 `agent-core`로 묶어서 다른 팀원들도 들어와서 볼 수 있게 세팅하자."
- **요약:** 절대 건드리면 안 되는 금고(root)와, 자유롭게 써야 하는 작업 책상(agent-admin)의 용도에 맞춰 소유자를 다르게 부여한 것입니다.
