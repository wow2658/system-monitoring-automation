### 항목 2. 시스템 관제 자동화(`monitor.sh`) 기술적 구현 및 선택 근거

#### **Q2-1. 프로세스와 포트 식별에 사용한 명령어와 선택 이유**

- **프로세스 식별 (`pgrep -f agent-app` 또는 `pgrep -x nginx`)**
  - **이유 (Why):** 전통적인 `ps -ef | grep` 방식은 grep 명령어 자체의 프로세스까지 결과에 출력되어 추가적인 필터링(`grep -v grep`)이 필요합니다. 반면 `pgrep`은 프로세스 테이블에서 이름만으로 순수 PID만 빠르고 정확하게 추출할 수 있어 스크립트의 효율성과 안정성이 높습니다.
- **포트 식별 (`ss -tulnp | grep ":15034"`)**
  - **이유 (Why):** 과거에 쓰이던 `netstat`은 현재 리눅스 배포판에서 공식적으로 지원 종료(Deprecated)된 레거시 도구입니다. `ss` 명령어는 커널 내부에서 직접 네트워크 상태를 읽어오기 때문에 속도가 훨씬 빠르며, 최신 리눅스 환경의 표준이므로 이를 선택했습니다.

#### **Q2-2. 시스템 자원 파싱 방식과 로그 포맷 고정 이유**

- **추출/파싱 방식:** `top`, `free`, `df` 명령어의 결과를 `|` (파이프)로 넘겨 `awk`와 `sed` 도구로 가공했습니다.
  - **이유 (Why):** 시스템 명령어의 원본 출력 결과에는 불필요한 헤더나 문자열이 섞여 있어 크기 비교(예: CPU > 20)를 할 수 없습니다. `awk`를 통해 특정 열(Column)의 순수 숫자 데이터만 뽑아내어 수학적 임계값 비교가 가능하도록 만들었습니다.
- **로그 포맷 고정 (`[시간] PID:... CPU:..% MEM:..% DISK:...%`)**
  - **이유 (Why):** 사람이 읽기 편할 뿐만 아니라, 이후 '장애 사후 분석(과제 2)' 시 엑셀이나 데이터 분석 툴(또는 `report.sh`)에서 공백(Space)을 기준으로 데이터를 쉽게 잘라내어(Split) 통계를 낼 수 있도록 정형화된 Key-Value 형태로 고정했습니다.

#### **Q2-3. 소유자/실행자 권한 정책 분리(RBAC) 구현 방식**

- **구현:** `chown agent-dev:agent-core` 설정 후 `chmod 750` 부여.
- **이유 (Why):** 최소 권한의 원칙(Least Privilege)을 지키기 위함입니다.
  - **소유자(`agent-dev`, 7=rwx):** 개발자는 스크립트를 수정하고 테스트할 수 있습니다.
  - **그룹(`agent-core`, 5=r-x):** `agent-admin`(cron 실행자)은 이 그룹에 속해 있으므로 스크립트를 '읽고 실행'할 수 있어 모니터링은 정상 동작합니다. 수정은 불가합니다.
  - **기타 사용자(0=---):** 외부인이나 다른 테스트 계정은 접근 자체를 차단하여 악성 코드 삽입을 방지합니다.

#### **Q2-4. 용량 기반 로그 관리(10MB/10개) 구현 방식**

- **구현:** 쉘 스크립트 내부에 복잡한 파일 제어 로직을 짜지 않고, OS 내장 표준 도구인 `logrotate` 설정 파일에 `size 10M`, `rotate 10` 옵션을 명시하여 구현했습니다.
- **이유 (Why):** 파일이 커질 때 스크립트로 직접 자르고 지우는 로직을 구현하면, 프로세스가 로그를 쓰고 있는 찰나의 순간에 충돌(Lock)이 발생해 로그가 유실될 위험이 큽니다. 리눅스 표준인 `logrotate`를 사용하는 것이 가장 안전하고 현업에서 쓰이는 권장 방식입니다.

### 항목 3. 시스템 보안 및 운영 정책에 대한 기술적 근거

#### **Q3-1. SSH 포트 변경과 Root 접속 차단이 보안에 효과적인 이유 (위협 모델 관점)**

- **SSH 포트 변경 (22 ➔ 20022):**
- **이유 (Why):** 인터넷에 노출된 서버는 24시간 내내 22번 포트를 노리는 봇넷(Botnet)의 자동화된 무차별 대입 공격(Brute-force)을 받습니다. 포트를 비표준 대역으로 바꾸는 것만으로도 단순 스캐너에 의한 무의미한 공격 트래픽과 로그 폭탄을 99% 이상 차단할 수 있습니다. 하지만 사실 이것도 뻔해서 드라마틱한 보안 효과는 아닙니다.
- **Root 원격 접속 차단 (`PermitRootLogin no`):**
- **이유 (Why):** `root`는 리눅스 시스템의 절대 권한자이며 이름이 고정되어 있습니다. Root 접속이 열려있으면 해커는 '비밀번호' 하나만 맞추면 됩니다. 하지만 이를 차단하면 해커는 존재하는 '일반 계정명'과 '비밀번호'를 모두 알아내야 하고, 접속 후에도 권한 상승(Privilege Escalation) 취약점을 뚫어야 하므로 공격 난이도가 기하급수적으로 상승합니다.

#### **Q3-2. `api_keys`와 로그 디렉토리를 `agent-core`로 제한한 이유 (최소 권한 원칙)**

- **설명:** 최소 권한 원칙(Principle of Least Privilege)이란 사용자나 프로세스가 목적을 달성하는 데 필요한 '딱 그만큼의 권한'만 부여하는 보안 설계의 핵심입니다.
- **이유 (Why):** `api_keys`는 외부 서비스와 연동되는 치명적인 크리덴셜(비밀키)이며, 로그 디렉토리는 시스템의 모든 행위가 기록되는 감사(Audit)의 핵심 증거입니다. 이를 핵심 운영 그룹인 `agent-core`에만 허용함으로써, 만약 권한이 낮은 QA/테스트 계정(`agent-test`)이 해킹당하더라도 해커가 비밀키를 탈취하거나 자신의 침입 로그를 삭제(증거 인멸)하는 것을 원천적으로 차단하기 위함입니다.

#### **Q3-3. "경고(Warning)"와 "종료(Exit)" 기준을 분리한 운영상의 이유**

- **설명:** 프로세스 사망/포트 닫힘은 즉각 스크립트를 종료(`exit 1`)시키지만, 방화벽 비활성이나 CPU/MEM 임계치 초과는 경고 메시지만 남기고 스크립트를 계속 실행합니다.
- **이유 (Why):** '서비스 가용성(Availability)'과 '관측성(Observability)' 확보 때문입니다. 포트가 닫힌 것은 서비스가 완전히 죽은 '장애(Down)' 상태지만, CPU가 90%를 치는 것은 느려졌을 뿐 아직 서비스가 동작 중인 '지연(Degraded)' 상태일 수 있습니다. 이때 모니터링 스크립트마저 강제로 멈춰버리면, 운영자는 가장 중요한 순간(장애 직전 피크 타임)의 시스템 지표를 수집하지 못해 사후 원인 분석(Postmortem)을 할 수 없게 됩니다.

#### **Q3-4. 리다이렉션 기호 `>` 와 `>>` 의 차이와 로그 누적 원리**

- **설명:** \* `>` (덮어쓰기/Overwrite): 파일의 처음부터 내용을 새로 씁니다. 기존 데이터는 모두 날아갑니다.
  - `>>` (이어쓰기/Append): 기존 파일 내용의 가장 마지막 줄(EOF) 뒤에 새로운 내용을 추가합니다.
- **이유 (Why):** `monitor.sh`는 `cron`에 의해 1분마다 주기적으로 실행됩니다. 이때 `>` 기호를 사용하면 매분마다 과거의 로그가 삭제되고 방금 측정한 1분 전의 데이터 딱 한 줄만 남게 됩니다. 서버의 상태 변화 추이(Timeline)를 기록하여 장애 시점의 증거로 삼기 위해서는 반드시 `>>` 를 사용하여 과거 데이터가 유실되지 않고 누적되도록 설계해야 합니다.

### 항목 4. Nginx 환경 변경 및 트러블슈팅 대응 방안

#### **Q4-1. 모니터링 대상이 Nginx로 변경될 경우 `monitor.sh` 핵심 수정 포인트와 근거**

**1. 프로세스 (Process)**

- **변경:** `pgrep -f agent-app` ➔ `pgrep -x nginx` (또는 `systemctl is-active nginx`)
- **이유 (Why):** Nginx는 트래픽 처리를 위해 하나의 Master 프로세스와 여러 개의 Worker 프로세스를 생성하는 '다중 프로세스' 구조로 동작합니다. 단순 문자열 검색(`-f`)을 사용하면 사용자가 편집기에서 `vi nginx.conf`를 켜놓은 것까지 프로세스로 오탐지할 수 있습니다. 따라서 정확히 'nginx'라는 이름의 실행 프로세스만(`-x`) 타겟팅해야 합니다.

**2. 포트 (Port)**

- **변경:** `15034` ➔ `80` (HTTP) 또는 `443` (HTTPS)
- **이유 (Why):** 15034는 기존 실습용 파이썬 앱(`agent-app`)이 사용하도록 임의로 할당된 커스텀 포트입니다. 반면 Nginx는 전 세계 표준 웹 서버이므로, 웹 서비스 기본 규약에 따라 HTTP 통신에는 80번, 보안이 적용된 HTTPS 통신에는 443번 포트를 사용하도록 설계되어 있습니다. 웹 서버의 정상 구동 여부를 파악하려면 반드시 이 표준 포트의 바인딩 상태를 감시해야 합니다.

**3. 로그 (Log)**

- **변경:** `/var/log/agent-app/*` ➔ `/var/log/nginx/access.log` 및 `error.log`
- **이유 (Why):** 기존 커스텀 앱은 개발자가 지정한 임의의 폴더에 단일 로그를 남겼습니다. 하지만 Nginx는 리눅스 패키지 표준 규약을 따르므로 기본적으로 `/var/log/nginx/` 경로에 로그를 적재합니다. 또한, 정상적인 사용자 방문 기록(`access.log`)과 서버 내부 장애 기록(`error.log`)을 물리적으로 엄격하게 분리하여 관리하므로, 모니터링 타겟 경로 역시 이 두 파일로 변경되어야 합니다.

**4. 임계값 (Threshold) - [핵심 차별화 포인트]**

- **변경:** CPU/MEM 의존도 기준 하향 ➔ 동시 접속 커넥션 수(TCP ESTABLISHED) 관제 로직 추가
- **이유 (Why):** 웹 서버(Nginx) 장애의 90% 이상은 CPU의 연산 폭주가 아니라, 과도한 트래픽 유입으로 인한 **네트워크 I/O 병목 및 소켓(포트) 고갈**에서 발생합니다. 따라서 CPU 사용률보다 '현재 연결된 클라이언트 수'가 Nginx의 상태를 대변하는 훨씬 중요한 지표가 됩니다.

#### **Q4-2. 프로세스는 살아있는데 포트가 열리지 않는 현상의 원인**

**원인 후보 1: 권한 부족 (가장 유력한 정답)**

- **이유 (Why):** 리눅스 보안 정책상 1024번 이하의 Well-known 포트(80, 443 등)는 일반 사용자 계정(`agent-admin`) 권한으로 열 수 없습니다. 오직 `root` 권한으로만 소켓 바인딩이 가능하므로, 일반 계정으로 런타임을 실행하면 프로세스는 메모리에 올라가지만 포트 바인딩 단계에서 권한 거부(Permission Denied) 에러가 발생합니다.

**원인 후보 2: 바인딩 IP 설정 오류**

- **이유 (Why):** 앱(웹 서버) 설정 파일에서 외부 접속을 허용하는 `0.0.0.0`이 아니라 내부 로컬 통신용인 `127.0.0.1(localhost)`로 서버를 띄웠을 수 있습니다. 이 경우 서버 내부의 프로세스는 정상적으로 살아있지만, 외부 네트워크망 시점에서는 포트가 닫힌 것(Connection Refused)으로 관측됩니다.

**원인 후보 3: 초기화 지연 (무한 대기/Hang 현상)**

- **이유 (Why):** 대다수의 백엔드 애플리케이션은 '프로세스 실행 ➔ DB 커넥션 등 외부 자원 연결 ➔ 포트 개방(Listen)' 순으로 부트 시퀀스가 진행됩니다. 만약 연동된 DB가 죽어있거나 네트워크 타임아웃 대기 상태에 빠지면, 프로세스는 켜져 있지만 포트를 열지 못하고 무한 대기 상태가 됩니다.

#### **Q4-3. 로그 급증으로 인한 디스크 풀(Disk Full) 장애 발생 시 대응 방안**

**1. 단기 대응 (즉각적인 장애 복구)**

- **대응:** `rm -rf` 명령어로 로그 파일을 삭제하지 않고, **`> /var/log/nginx/error.log` (또는 `cat /dev/null > 로그파일`) 명령어를 사용하여 파일의 내용만 0바이트로 비웁니다(Truncate).**
- **이유 (Why) [★매우 중요]:** 리눅스는 실행 중인 프로세스(Nginx)가 특정 로그 파일을 '사용 중(Open)'일 때 해당 파일 자체를 `rm`으로 삭제해버리면 프로세스가 파일 연결고리(File Descriptor)를 놓지 않습니다. 그 결과 **디스크의 실제 여유 공간이 반환되지 않고 여전히 100% 가득 찬 상태를 유지하여 서버가 다운**됩니다. 반드시 파일 껍데기(Inode)는 그대로 두고 내부 데이터만 비워내야 즉각적으로 디스크 용량이 확보됩니다.

**2. 중기 대응 (재발 방지 시스템화)**

- **대응:** `logrotate` 설정을 기간(Daily 등) 기준에서 **임계 용량(Size, 예: 100M) 기준**으로 변경하고, 압축(`compress`) 옵션을 활성화합니다.
- **이유 (Why):** 운영자가 24시간 서버를 모니터링하며 수동으로 로그를 비울 수는 없습니다. 따라서 OS 내부 스케줄러가 특정 로그 파일이 임계 용량에 도달하면 자동으로 파일을 분리(Rotate)하고 이전 로그를 압축하도록 시스템에 위임해야, 예기치 않은 트래픽 폭주에도 디스크 가용성을 안전하게 방어할 수 있습니다.
